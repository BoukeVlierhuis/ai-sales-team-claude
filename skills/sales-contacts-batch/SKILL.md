# Sales Contacts — Batch Mode

You run the **sales-contacts** procedure (commercial-focus decision-maker intelligence) over a list of URLs read from a `.txt` file, sequentially, in a single conversation. Per prospect you produce a `DECISION-MAKERS-{slug}.md`. At the end you write one `BATCH-SUMMARY-{date}.md` with a comparison table.

## Phase 0: Load Seller Context

**Requires `.sales/` (project-local seller config).** If missing, error: `"No seller config found. Run /sales init to set one up."`

**Requires `--proposition=<slug>` argument.** The same proposition applies to every URL in the batch. If missing, list available propositions from `.sales/propositions/*.md` and error: `"Which proposition? Available: <slug list>. Re-run with --proposition=<slug>."` If slug unknown, error: `"Proposition '<slug>' not found in .sales/propositions/. Available: <slug list>."`

**Files to load:**
- `.sales/identity.md` — seller name and primary sender persona; used to scope which decision-maker titles map to which sender for the outreach hand-off
- `.sales/propositions/<slug>.md` — proposition's Target Persona; used to bias the decision-maker search toward titles that buy THIS proposition

**Use seller context.** Every per-prospect `DECISION-MAKERS-{slug}.md` is generated by the same procedure as `/sales contacts` and inherits its scoring rules: "Title Match" is scored against the proposition's Target Persona from `.sales/propositions/<slug>.md`, not a generic ICP. Surface the matching persona name in each report.

Every generated file (both per-prospect `DECISION-MAKERS-{slug}.md` and the final `BATCH-SUMMARY-{date}.md`) starts with this header block:

    Seller: <identity.company>
    Proposition: <slug> — <name>
    ICP: .sales/icp.md
    Generated: <date>

## Invocation

```
/sales-contacts-batch <path-to-urls.txt> --proposition=<slug>
```

The argument is **always a path to a `.txt` file**. Inline URL lists are not supported — point users to `/sales contacts <url> --proposition=<slug>` for one-off research.

## Input Format (`urls.txt`)

- One URL per line
- Lines starting with `#` are comments and skipped
- Empty lines are skipped
- Whitespace around URLs is trimmed
- Both bare hostnames (`datavisual.nl`) and full URLs (`https://www.uwv.nl/`) are accepted

Example:
```
# Q2 prospects — round 1
https://www.uwv.nl/
datavisual.nl
acme-software.com
# foo.io   ← skip for now
```

If the file is missing, empty, or has no valid URLs after stripping comments, stop and report the issue — do not proceed.

## Procedure

For each URL in the file, in order:

1. **Print progress line** to the terminal: `[N/Total] {url}  ⏳ ...`
2. **Run the full sales-contacts procedure** for that URL (Phase 1–5 of the `sales-contacts` skill: team page analysis, LinkedIn research focused on commercial roles, commercial org chart, buying committee classification, personalization anchors, scoring, multi-threading strategy) — using the seller context loaded in Phase 0 to bias scoring toward the selected proposition's Target Persona.
3. **Write** `DECISION-MAKERS-{slug}.md` in the current working directory using the slug rules from `sales-contacts` (lowercase, strip TLDs and legal suffixes, hyphenate spaces, keep `[a-z0-9-]`).
4. **Update the progress line** with the result:
   - `✓ score X/100 → DECISION-MAKERS-{slug}.md` on success
   - `✗ {short reason}` on failure (404, no contacts found, rate-limited, etc.)
5. **Continue to the next URL** even if this one failed. One failure must not stop the batch.
6. **Pause briefly between URLs** (one short delay) to avoid hammering WebSearch/WebFetch. Do not retry failures inside the batch — collect them and report at the end.

## Output: BATCH-SUMMARY-{YYYY-MM-DD}.md

After all URLs are processed, write a single summary file in the current directory. Filename uses today's date in `YYYY-MM-DD` format. If a file with that name already exists, overwrite it.

Format:

```markdown
# Sales Contacts Batch Summary — {YYYY-MM-DD}

Proposition: <slug> — <name>

Generated: <date>
```

## Scope and Limits

- **Designed for 5–30 URLs per run.** Below 5: use `/sales contacts <url> --proposition=<slug>`. Above 30: split into multiple `.txt` files or use a shell loop with separate sessions.
- **Sequential by design.** This skill does not parallelize. Failures are easy to inspect because each prospect is processed in order and tool calls remain visible per prospect.
- **One proposition per batch.** The `--proposition=<slug>` flag applies to every URL in the file. If you want to compare the same prospects against multiple propositions, run the batch once per proposition.
- **Commercial focus inherited.** This skill reuses the `sales-contacts` procedure verbatim, which is permanently scoped to commercial roles (CRO/VP Sales, BD, RevOps, AE/SDR managers). Non-commercial contacts are out of scope.
- **No retry logic.** If a prospect fails, log it and continue. Retry by re-running with a smaller `.txt` file containing only the failed URLs.

## Error Handling

- **Missing seller config or proposition:** See Phase 0 error rules above. Do not start processing URLs until both are resolved.
- **Missing file:** Stop immediately, report the path that was not found.
- **Empty or all-comment file:** Stop, report "no URLs to process".
- **Per-URL failures:** Log to the failed-URLs table in the batch summary, continue with the next.
- **Total batch failure (e.g. rate-limit hit hard mid-run):** Still write the partial `BATCH-SUMMARY` with whatever was completed, and clearly mark the remaining URLs as "not attempted".

## Cross-Skill Integration

- Each `DECISION-MAKERS-{slug}.md` is consumable by `/sales outreach`, `/sales prep`, and `/sales prospect` exactly as if generated by `/sales contacts` directly — and inherits the same `--proposition=<slug>` so downstream skills can pick it up.
- Suggest follow-ups in the terminal summary: "Top 3 prospects by score — consider running `/sales outreach <url> --proposition=<slug>` for each."
